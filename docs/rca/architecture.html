Parallel context fetch

Stage 04 is the first big latency win. The primer is assembled by parallel lookups, so wall time follows the slowest request instead of the sum of all requests.

_build_primer

Primer assembly

Input envelope

log_window, error_class, and build_meta enter as the seed context.

Concurrent fan-out

Service config, Jira facts, Jenkins node facts, and retrieval snippets are launched in parallel.

One primer, one message

The final primer is injected as the system message used for agent iteration 0.

03

Agent iteration loop

The reasoning core uses OpenAI gpt-4o function-calling. Iteration 0 gathers broad context, later iterations drill into specifics, and the server refuses duplicate tool calls.

Tool loop
0 iter
Parallel fan-out

read_runbook(class), get_jenkins_job_config, list_job_flows, repo_read_file, and aws_describe can all land in the same assistant turn and execute concurrently.

1 iter
Drill helpers

repo_find_function, repo_search, jenkins_node_info, and rag_search sharpen the hypothesis based on results from the first sweep.

2 iter
Confirm and emit JSON

Final output uses response_format=json_object with required keys for summary, stage, class, cause, evidence, and fix.

↻ gate
Gate trip path

If a LangGraph gate fires, corrective_reiter performs one extra tool iteration plus one forced-final pair. Retry budget is 1 and cost is usually about $0.10 extra.

✓ done
Final RCA JSON

The agent exits, validators run, signals are recorded, and the response moves to persistence and delivery.

Hard caps

25 tool calls per RCA
180s wall-clock budget
$5 panic killswitch
3 iterations typical

Defenses

  • Tool dedup cache uses a fingerprint of name plus args.
  • Duplicate call attempts return an explicit server-side error.
  • text_tool_calls rescue handles the model writing tool calls as text.
  • Cap or wall-clock breach triggers forced finalization automatically.
04

RAG flow with Postgres and pgvector

Retrieval uses an HNSW cosine index over text-embedding-3-small vectors. Every RCA gets retrieval context injected at iteration 0.

Auto-injected memory

RAG indexing

offline · cron 2h

Sync script self-pulls the repo, indexes docops and audits, then restarts the service.

1
Discover sources docops/runbooks/*.md, docops/job_flows/*.md, docops/*.md, and /var/log/bbctl-rca/<uuid>.json that already match RCA shape.
2
Chunk on H2 splits Target around 2K chars with 200-char overlap and section headings preserved for retrieval signal.
3
Embed in batches OpenAI text-embedding-3-small with batches up to 100, stable on retry.
4
Upsert rca_chunks Rows carry id, source metadata, chunk index, text, vector[1536], and score.
5
HNSW rebuild Cosine index with ef_construction=200; query latency is about 5 ms at 500 chunks.

RAG query

online · per RCA

Runs concurrently with primer fan-out and lands inside the system prompt as a ## retrieved.rag block.

1
Anchor query slice Use the last Error: line plus 500 chars of context, or the last 1.5K chars if no error marker exists.
2
Embed cache check sha256(query) hits diskcache for 24 hours; misses trigger the OpenAI embedding call.
3
HNSW cosine search Top-5 retrieval, filtered per R3.1 so known classes prefer runbook plus audit while unknown also includes general docs.
4
Format hits Render a bulleted retrieval block with source path, score, and excerpt length around 500 chars.
5
Inject into primer Past RCA hits often score 0.71+, above runbooks at 0.4-0.6, which makes retrieval act like semantic memory.
05

LangGraph state machine

The graph replaces nested branching with routable validation nodes. Adding a new gate now means adding a node and conditional edges instead of extending a fragile if-stack.

Lazy import safe

Gate sequence

START Enter the graph after the agent produces a candidate RCA.
_check_schema Missing summary, stage, class, cause, evidence, or suggested fix?
_check_runbook_fetched Class is in MANDATORY_RUNBOOK but the runbook was never read.
trip -> corrective_reiter

Append the gate reason, perform one more tool iteration, force a JSON answer, and decrement retry_budget.

_check_primary_secondary jenkins_agent_offline must identify PRIMARY or SECONDARY when relevant.
_check_compliance_hallucination Reject compliance claims that contradict pre-fetched Jira truth.
END Validated RCA moves to final hard validators and audit logging.
Loopback: if retry_budget > 0, corrective_reiter re-enters the gate path beginning at _check_runbook_fetched.

field purpose

read_runbooksSet of runbook names fetched during the agent loop.
messagesOpenAI history so gates can append corrective instructions.
client / model / tools / tool_dispatchPlumbing passed into corrective_reiter for the retry turn.
failure_signalsAccumulated telemetry for audit and dashboards.
retry_budgetDefaults to 1; each gate-triggered retry decrements it.

06

Three-layer slave-ID placeholder defense

The goal is to prevent bbctl shell <slave-instance-id> from ever reaching an operator. Each layer should reduce the amount of cleanup the next layer has to do.

Operator safety
Layer 1 · cooperative

Primer pre-fetch

When the log mentions slave-N, the system calls the Jenkins node API and injects a ## jenkins.node block with the real instance_id before iteration 0 starts.

Layer 2 · cooperative

Mandatory prompt rule

rca_common.md explicitly says that a log containing slave-N must trigger jenkins_node_info, supported by examples and anti-patterns.

Layer 3 · mandatory

Server gate

After final JSON generation, the server scans commands for <slave-instance-id>, substitutes the real value if known, or drops the command and appends an ops-contact fallback.

Cost model

The embedding layer is cheap. Most of the spend sits in the reasoning loop, with retries and corrective passes adding small but noticeable increments.

Production mean: $0.30
$0.00003Embed one anchor query slice.
$0.0001Embed one chunk during re-index.
$0.07Re-index the full docops set at roughly 230 chunks.
$0.10Re-index 100 newly added audits.
$0.20-$0.30Standard agent-path RCA when gates pass.
+$0.05-$0.10One gate-triggered retry pair.
+$0.05Schema-completion retry.
