3. RCA algorithm — the agent loop

INITIAL STATE
─────────────
  messages = [ system_prompt + 3-block primer,
               user_kickoff="Begin the RCA. Use tools." ]
  tools    = [ 16 OpenAI function-calling schemas ]

LOOP (max 25 iters, 180s wall clock, $5 hard kill)
─────────────────────────────────────────────────
  iter = 0
  while iter < 25:
      response = POST openai/chat/completions(messages, tools, tool_choice="auto")
      msg      = response.choices[0].message

      if msg.tool_calls:                          # LLM wants more data
          for tc in msg.tool_calls:               # parallel or single
              result = dispatch(tc.name, tc.args) # call local Python or external API
              messages.append({"role":"tool", "tool_call_id":tc.id, "content":result})

          # dedup: same (name, args) called twice within RCA
          #   1st repeat → return cached + soft warning
          #   2nd+ repeat → ERROR ONLY, no data (force LLM to pivot)

          iter += 1
          continue

      else:                                       # finish_reason=stop, content=JSON
          rca = parse_json(msg.content)
          break

POST-PROCESSING
───────────────
  validate_and_fix(rca, service, service_lookup)  # regex strip hallucinations
  audit.record(rca)
  return rca to caller
Key insight: The LLM decides WHAT to call. The server decides nothing — it just executes the calls, appends results, asks again. Every decision (which file, which AWS API, what conclusion) comes from the LLM reading the conversation history + tool schemas + system prompt.

4. What goes in the prompt (no spoon-feeding)

System message (~3000 tokens, sent every iter)

SectionPurposeSpoon-feed?
Role + boot contextWhat LLM is + what 3 blocks it gets No — just the contract
Method (backwards-scan, stage→helper, chain-walk) HOW to investigate, not WHAT to conclude No — recipe, not answer
Stage→helper convention table 9 stage names → helper file paths No — convention, LLM still reads
Anti-hallucination + value provenance Banned defaults (8080, /admin/version, gps.log) No — negative rules
Output schema Required JSON keys No — format only

User message (3 blocks per RCA, max ~5000 tokens)

## build_meta
- job: stagger-prod-plus-devops-test
- build: 46
- service: test-supply-wrapper-nonweb
- result: FAILURE
- url: https://jenkins-stagger.jinka.in/job/...

## service.lookup
{
  "aws_account": "zinka",
  "aws_region": "ap-south-1",
  "target_port": 8080,
  "rule_arn": "arn:aws:elasticloadbalancing:ap-south-1:735317561518:...",
  "filebeat_log_path": "/var/log/blackbuck/test-supply-wrapper-nonweb.log*",

## log_window (sanitized — tail of build log, error lives at the bottom)
<last 30K chars of the Jenkins log>
What's NOT sent: error_class, failed_stage, runbook content, Jira ticket data, GitHub commits, AWS state, any file content. The LLM fetches every one of these via tool calls. That's what makes it agent-mode, not spoon-feed.

5. The 16 tools the LLM can call

DomainToolWhat it doesSource
Jira jira_get_ticket(key) Fetch ticket fields + custom_fields Atlassian REST
jira_search(jql, max) JQL search for clone-chain discovery Atlassian REST

repo_search(repo, query, max) ripgrep across local clone Local FS
repo_find_function(repo, name) Locate function/helper (Jenkins vars/ convention) Local FS
repo_recent_commits(repo, n) Recent commits on local clone git log
Jenkins get_jenkins_job_config(job) scriptPath + scm + inline script Jenkins REST
Runbook list_runbooks() List 10 per-class .md files + summaries /opt/bbctl-rca/docops/runbooks/
read_runbook(name) Full markdown content Same
AWS aws_describe(service, operation, params, account, region) Any read-only AWS API (Describe / Get / List / Lookup / Search / Show / Estimate) boto3 + STS AssumeRole

SSM SendCommand intentionally NOT in this list. Per Option C decision, bbctl-rca never logs into instances. For service-side detail (e.g., "why is the service unhealthy"), the LLM tells the operator to use bbctl shell <instance_id> themselves.

8. Why this is targeted-read, NOT whole-repo scan

Whole-repo scan (NOT what we do)

find /opt/bbctl-rca/repos -name "*.groovy"
  → 35+ files in vars/
  → 50+ files in src/
  → dozens in resources/
read all → embed all → ask LLM
  → 500K+ tokens
  → $5+ per RCA
  → LLM lost in noise

What bbctl-rca actually does

log says: stage 'Deploy Prod+1' failed
  → LLM applies stage→helper table:
       Deploy Prod+1 → vars/deployProdPlusOne.groovy
  → LLM calls repo_read_file on JUST that file
  → That file says: nonwebdeploy(...)
  → LLM reads vars/nonwebdeploy.groovy
  → That file says: libraryResource 'scripts/non_web_healthy.sh'
  → LLM reads resources/scripts/non_web_healthy.sh
TOTAL: 4 files, ~5K tokens, $0.22

How LLM picks files without scanning

  1. Convention table in system prompt — 9 stages → 9 helper paths. LLM doesn't search; it knows.
  2. Chain-walk rule — outer file always names inner file. LLM reads outer → sees name → reads next.
  3. libraryResource resolution — libraryResource 'X' always means resources/X on disk. Deterministic.
  4. repo_search fallback — only if 1–3 above don't apply. Almost never used for health_check / compliance / java_runtime / aws_limit classes because the convention covers them.

9. Three layers of anti-hallucination defense

LayerWhat it doesExample caught
1. Prompt rules "Every concrete value in final JSON MUST come from a tool result in this RCA's message history. Banned defaults: port 8080, /admin/version, /var/log/blackbuck/gps.log." LLM sees the rule and uses real values from aws_describe response — works ~70% of the time.
2. Runbook MANDATORY calls health_check.md drill plan marks DescribeTargetGroups as MANDATORY in iter 0 — so real Port + HealthCheckPath are in LLM context before it drafts the fix. LLM has the real values structurally available, less likely to default.
3. Server validator bbctl_rca/value_validator.py runs after LLM emits JSON. Regex strips (?<![\d])8080(?![\d]) → service.lookup.target_port; /admin/version → service.lookup.health_check_path; gps.log → filebeat_log_path when service != gps. Mechanical, always runs. Catches whatever the LLM still gets wrong. Substitutions recorded in result["validator_notes"][].
